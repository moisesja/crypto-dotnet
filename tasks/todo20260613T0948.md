# Issue #2 — Expose the BBS `header` parameter on `IBbsCryptoProvider`

## Evaluation (verified against the code)

**Claim:** `IBbsCryptoProvider` hardcodes the BBS signature `header` to empty at the FFI
boundary, so a consumer (`dataproofs-dotnet` `bbs-2023`) cannot bind the mandatory-disclosure
group and a malicious holder can drop a mandatory message and still pass `VerifyProof`.

**Verified true. Findings:**

1. `src/NetCrypto/DefaultBbsCryptoProvider.cs` passes `ReadOnlySpan<byte>.Empty, 0` for the
   header in all four ops:
   - `Sign` → `bbs_sign(privateKey, pk, Empty, 0, ...)` (line ~112)
   - `Verify` → `bbs_verify(publicKey, Empty, 0, ...)` (line ~137)
   - `DeriveProof` → `bbs_proof_gen(pk, sig, Empty, 0, nonce, nonce.Length, ...)` (line ~191)
   - `VerifyProof` → `bbs_proof_verify(pk, proof, len, Empty, 0, nonce, nonce.Length, ...)` (line ~226)

2. The existing `nonce` param IS the BBS presentation header (`ph`), not the signature
   `header` — confirmed by the FFI mapping (`nonce` → `ph_ptr/ph_len`). The issue is correct:
   the fix must add a NEW `header`, not reuse `nonce`/`ph`.

3. **The native side already fully supports AND consumes the header — no Rust/binary change
   is required.** `native/zkryptium-ffi/src/lib.rs` decodes `header_ptr/header_len` in
   `bbs_sign`/`bbs_verify`/`bbs_proof_gen`/`bbs_proof_verify` and threads `header_opt` into
   zkryptium's `sign`/`verify`/`proof_gen`/`proof_verify`. The Rust unit tests already
   round-trip with non-empty headers (`b"test-header"`, `b"credential-header"`). The current
   passing managed NativeFFI tests already call these entry points with the header arg pair
   (empty), proving the shipped binaries' ABI matches. => **Pure managed-layer fix.**

4. Only implementer of `IBbsCryptoProvider` in-repo is the sealed `DefaultBbsCryptoProvider`.
   All in-repo callers (samples, tests) use POSITIONAL args; no `nonce:` named-arg callers
   exist. Version is `1.0.0-preview.1` (pre-1.0 prerelease) → the minor API change is OK.

**Scope boundary:** NetCrypto only *surfaces* the header. The actual `bbs-2023` binding
(`bbsHeader = SHA-256(proofHash) ‖ SHA-256(mandatoryHash)`) is `dataproofs-dotnet`'s job.
Do NOT implement that binding here — only the parameter passthrough + tests proving the
header is committed by sign/verify and derive/verify.

## Design decisions

- **Optional parameter, not overloads.** Add `ReadOnlySpan<byte> header = default` as the last
  parameter on each of the 4 methods (interface + impl). `default` is a legal optional default
  for a ref struct, preserves current empty-header behavior, and is what the issue + downstream
  expect. Declare the default identically (`= default`) on BOTH interface and class to avoid the
  C# "interface vs concrete default" divergence gotcha. (Overloads/DIM rejected: more surface,
  unnecessary pre-1.0 with a single sealed impl.)
- **Rename `nonce` → `presentationHeader`** on `DeriveProof`/`VerifyProof` (interface + impl +
  XML docs). Reason: leaving `nonce` next to a new `header` is actively misleading; the param
  truly is the BBS `ph`. Safe: no named-arg callers in-repo. (If the maintainer prefers minimal
  churn, this rename can be dropped without affecting the security fix.)
- The implementation change is literally `header, (nuint)header.Length` in place of
  `ReadOnlySpan<byte>.Empty, 0` in each of the 4 FFI calls. No new validation needed (zkryptium
  handles header length).
