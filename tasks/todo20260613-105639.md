## Review

### Findings

1. Public malformed-key paths still leak backend exceptions instead of validating at the
   public boundary:
   - NSec-backed Ed25519/X25519 paths throw `System.FormatException` from `Key.Import` /
     `PublicKey.Import`.
   - BLS paths throw `Nethermind.Crypto.Bls+BlsException`.
   - NIST EC private-key paths can throw platform-specific backend `CryptographicException`
     before length/scalar validation; on macOS this also prints `CSSM_ModuleLoad()` errors.
   These violate PRD NFR-3's parameter-named `ArgumentException` / `ArgumentNullException`
   boundary.

2. `DeriveX25519PublicKeyFromEd25519` accepts invalid 32-byte Ed25519 public encodings.
   All-zero Ed25519 input returns X25519 public key `01..00`; `DeriveSharedSecret` later rejects
   that low-order key, but the converter can still mint an invalid `PublicKeyReference`.

3. Rust dependency audit reports `RUSTSEC-2026-0097` for transitive `rand 0.8.5` via
   `zkryptium 0.6.1`. Current code does not define a custom Rust logger or call `rand` directly,
   so exploitability appears low here, but the dependency is still on an affected version.

### Refuted / Not Findings

- BBS `VerifyProof` malformed disclosure metadata did not produce a proof-verification bypass in
  the local native-present probe. A valid proof verified normally; duplicate, out-of-range,
  negative, and empty revealed-index inputs returned `false`.
- AES-CBC-HMAC verifies the tag with `FixedTimeEquals` before CBC decrypt, so no padding-oracle
  path was found.

### Verification

- `dotnet test NetCrypto.sln --filter Category!=BbsAbsent` passed: 703 tests.

  loading NuGet vulnerability data.
- `cargo test` in `native/zkryptium-ffi` passed: 3 tests.

- `dotnet list NetCrypto.sln package --vulnerable --include-transitive` reported no vulnerable

- Local throwaway probe under `/private/tmp/netcrypto-audit-probe` reproduced findings 1 and 2
  and refuted the BBS proof-metadata bypass candidate.

## Resolution (preview.3, branch `fix/malformed-key-input-hardening`)

**Finding 1 — backend exceptions on malformed keys → FIXED.** Added `RawKeyGuard.RequireLength`
and applied up-front length validation at every backend hand-off:
- Ed25519: `DefaultCryptoProvider.SignEd25519`/`VerifyEd25519`, `DefaultKeyGenerator.RestoreEd25519`.
- X25519: `KeyAgreement`, `DeriveX25519SharedSecret`, `RestoreX25519`.
- NIST EC private key: `ImportEcPrivateKey` (chokepoint for Sign/DeriveSharedSecret/Restore) via
  `EcScalarByteLength` (P-256=32, P-384=48, P-521=66).
- BLS: `SignBls`/`RestoreBls` — length guard plus a `try/catch` mapping Nethermind's
  `BlsException` (incl. the valid-length/invalid-scalar case) to `ArgumentException(privateKey)`.
All now throw a parameter-named `ArgumentException`. (NIST valid-length/out-of-range D still
surfaces as the platform `CryptographicException` — in-contract, a genuine crypto failure, not a
malformed-length input; deliberately not normalized.)

**Finding 2 — invalid Ed25519 → X25519 conversion → FIXED.** `DeriveX25519PublicKeyFromEd25519`
now rejects the five canonical Curve25519 low-order u-coordinates (the all-zero key maps to u=1),
throwing `ArgumentException(ed25519PublicKey)`. Adversarially verified: algebraically constructing
inputs that target each forbidden u is rejected (0 bypasses), and 1000 real Ed25519 keys convert
without false rejection.

**Finding 3 — RUSTSEC-2026-0097 → DOCUMENTED (no in-tree fix).** Remediation is gated on a
`zkryptium` release that bumps `rand` past 0.8.x (the fix is in `rand 0.9`, a breaking change);
documented in `native/zkryptium-ffi/README.md`. CI continues to `--ignore` it with the
non-reachability justification.

**Test/contract changes.** Deleted the `KnownBackendDeviations` allow-list in
`InputValidationFuzzTests` (no more pinned NFR-3 breaches); added `KeyInputValidationTests`
(parameter-named assertions + low-order rejection + valid-key round-trip sanity); tightened the
PRD NFR-3 AC; recorded lesson L5.

**Verification.** Full native-present leg (`Category!=BbsAbsent`) green; adversarial sweep
(8 key types × 18 lengths × 3 content patterns × all byte-oriented entry points) found only
in-contract exception types — no `FormatException`/`BlsException`/`IOOR`/`NRE`/`Overflow` leaks.
